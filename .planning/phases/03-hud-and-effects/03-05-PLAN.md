2. In `build_hud_state()`, compute health bars by projecting world positions to screen space.

First, find how the view-projection matrix (MVP) is computed in app.rs. Look for `Matrix4::look_at` or `perspective` or `mvp`. The Camera + Screen produce the MVP used by all render passes.

For each alive unit with health < max_health, project its world position to screen:
```rust
let mut health_bars = Vec::new();
// Get the view-projection matrix
let mvp = /* compute or access the MVP matrix from camera/screen */;
let screen_w = self.screen.width as f32;
let screen_h = self.screen.height as f32;

for unit in self.unit_coordinator.units().iter().filter(|u| u.alive && u.health < u.max_health) {
    // World position: unit.movement.pos is WorldCoord, convert to f32 3D
    let world_x = unit.movement.pos.x as f32 / 128.0; // world to cell-space if needed
    let world_y = unit.movement.pos.y as f32 / 128.0;
    let world_z = /* height at unit position, lookup from landscape */;

    // Project to normalized device coords
    let world_pos = cgmath::Vector4::new(world_x, world_y, world_z + 1.5, 1.0); // +1.5 for bar above head
    let clip = mvp * world_pos;
    if clip.w <= 0.0 { continue; } // behind camera
    let ndc_x = clip.x / clip.w;
    let ndc_y = clip.y / clip.w;
    // NDC to screen coords (NDC is -1..1, screen is 0..w, 0..h)
    let sx = (ndc_x + 1.0) * 0.5 * screen_w;
    let sy = (1.0 - ndc_y) * 0.5 * screen_h; // flip Y

    // Only include if on-screen
    if sx >= 0.0 && sx <= screen_w && sy >= 0.0 && sy <= screen_h {
        health_bars.push(HealthBarEntry {
            screen_x: sx,
            screen_y: sy,
            health_fraction: unit.health as f32 / unit.max_health.max(1) as f32,
            bar_type: HealthBarType::Unit,
        });
    }
}
```

Note: The exact world-coordinate-to-3D-position transformation depends on how the terrain renderer maps coordinates. Check how person sprites are positioned (search for "sprite" in the render pass) to find the exact coordinate mapping. Adapt the same transform.

3. In `draw_hud()`, render health bars from HudState:
```rust
// === World-Space Health Bars ===
let bar_w = 32.0 * layout.scale_x;
let bar_h = 3.0 * layout.scale_y;
for hb in &hud_state.health_bars {
    let bx = hb.screen_x - bar_w / 2.0;
    let by = hb.screen_y - bar_h;
    // Background
    hud.draw_rect(bx, by, bar_w, bar_h, [0.0, 0.0, 0.0, 0.6]);
    // Fill (green->yellow->red based on fraction)
    let color = if hb.health_fraction > 0.5 { [0.1, 0.8, 0.1, 0.9] }
                else if hb.health_fraction > 0.25 { [0.8, 0.8, 0.1, 0.9] }
                else { [0.8, 0.1, 0.1, 0.9] };
    hud.draw_rect(bx, by, bar_w * hb.health_fraction, bar_h, color);
}
```

3. **Wire into combat events** -- In the combat module, find where damage is applied and units die. After the existing processing, collect EffectActions:
- On unit death: `EffectAction::SpawnAt { effect_type: 0x30 (DeathPuff), x, y, z, owner }`
- On melee hit: `EffectAction::SpawnAt { effect_type: 0x33 (HitSpark), x, y, z, owner }`
- On projectile impact: `EffectAction::SpawnAt { effect_type: 0x32 (BloodSpray), x, y, z, owner }`

The actual pattern depends on how combat is structured. If combat returns actions (like BuildingTickActions), add effect actions to the return. If combat modifies state directly, collect effect actions in a Vec and process after.

4. **Wire into building events** -- In the building module, find state transitions:
- Construction start -> `EffectAction::SpawnAt { effect_type: 0x50 (ConstructionDust), ... }`
- Building destroyed -> `EffectAction::SpawnAt { effect_type: 0x51 (DestructionCollapse), ... }`
- Building on fire -> `EffectAction::SpawnAt { effect_type: 0x52 (BuildingFire), ... }`

5. **Process effect actions in the tick loop.** In app.rs or wherever the tick results are processed, apply collected EffectActions:
```rust

            self.effect_pool.spawn(effect_type, x, y, z, owner);
        }
    }
}
```

6. **Call effect_pool.update_all() each tick** -- Add after the existing simulation_tick call:
