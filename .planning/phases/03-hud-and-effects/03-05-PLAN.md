2. In `build_hud_state()`, compute health bars by projecting world positions to screen space.

First, find how the view-projection matrix (MVP) is computed in app.rs. Look for `Matrix4::look_at` or `perspective` or `mvp`. The Camera + Screen produce the MVP used by all render passes.

For each alive unit with health < max_health, project its world position to screen:
```rust
let mut health_bars = Vec::new();
// Get the view-projection matrix
let mvp = /* compute or access the MVP matrix from camera/screen */;
let screen_w = self.screen.width as f32;
let screen_h = self.screen.height as f32;

for unit in self.unit_coordinator.units().iter().filter(|u| u.alive && u.health < u.max_health) {
    // World position: unit.movement.pos is WorldCoord, convert to f32 3D
    let world_x = unit.movement.pos.x as f32 / 128.0; // world to cell-space if needed
    let world_y = unit.movement.pos.y as f32 / 128.0;
    let world_z = /* height at unit position, lookup from landscape */;

    // Project to normalized device coords
    let world_pos = cgmath::Vector4::new(world_x, world_y, world_z + 1.5, 1.0); // +1.5 for bar above head
    let clip = mvp * world_pos;
    if clip.w <= 0.0 { continue; } // behind camera
    let ndc_x = clip.x / clip.w;
    let ndc_y = clip.y / clip.w;
    // NDC to screen coords (NDC is -1..1, screen is 0..w, 0..h)
    let sx = (ndc_x + 1.0) * 0.5 * screen_w;
    let sy = (1.0 - ndc_y) * 0.5 * screen_h; // flip Y

    // Only include if on-screen
    if sx >= 0.0 && sx <= screen_w && sy >= 0.0 && sy <= screen_h {
        health_bars.push(HealthBarEntry {
            screen_x: sx,
            screen_y: sy,
            health_fraction: unit.health as f32 / unit.max_health.max(1) as f32,
            bar_type: HealthBarType::Unit,
        });
    }
}
```

Note: The exact world-coordinate-to-3D-position transformation depends on how the terrain renderer maps coordinates. Check how person sprites are positioned (search for "sprite" in the render pass) to find the exact coordinate mapping. Adapt the same transform.

```rust
/// Spawn a visual effect for a spell impact at the given world position.
/// Called by the spell system (Phase 4) when a spell hits its target.
/// This function validates the mapping and spawns the correct effect type.
pub fn spawn_on_spell_impact(pool: &mut EffectPool, spell_type: u8, x: i32, y: i32, z: i32, caster_tribe: u8) {
    let effect_type = match spell_type {
        0x01 => 0x01, // Burn -> BurnFlame
        0x02 => 0x02, // Blast -> BlastProjectile
        0x03 => 0x03, // Lightning -> LightningBolt
        0x04 => 0x04, // Tornado -> TornadoVortex
        0x05 => 0x05, // Swamp -> SwampBubble
        0x06 => 0x06, // Flatten -> FlattenWave
        0x07 => 0x07, // Earthquake -> EarthquakeCrack
        0x08 => 0x08, // Erosion -> ErosionDrip
        0x09 => 0x09, // Volcano -> VolcanoEruption
        0x0A => 0x0A, // Firestorm -> FirestormRain
        0x0B => 0x0B, // AngelOfDeath -> AngelSwirl
        0x0C => 0x0C, // Shield -> ShieldBubble
        _ => return, // Unknown spell, no effect
    };
    pool.spawn(effect_type, x, y, z, caster_tribe);
}
```

Adapt the actual spell_type and effect_type constants to match what's defined in effects/types.rs from Plan 02. The key requirement: each spell type maps to a distinct visual effect type.
