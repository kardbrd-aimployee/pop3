```rust
/// Temporary compatibility: build a Unit view from pool data.
/// Will be removed once all consumers use pool directly.
pub fn unit_view(&self, handle: ObjectHandle) -> Option<Unit> {
    let obj = self.pool.get(handle)?;
    if let GameObjectData::Person(ref person) = obj.data {
        Some(Unit {
            id: handle as usize,
            model_type: obj.header.model_type,
            subtype: obj.header.subtype,
            tribe_index: obj.header.tribe,
            movement: person.movement.clone(),
            cell_x: person.cell_x,
            cell_y: person.cell_y,
            state: person.state,
            prev_state: person.prev_state,
            state_timer: person.state_timer,
            state_counter: person.state_counter,
            health: obj.header.health,
            max_health: obj.header.max_health,
            target_unit: person.target_unit.map(|h| h as usize),
            attacker_unit: person.attacker_unit.map(|h| h as usize),
            alive: person.alive,
            home_pos: person.home_pos,
            behavior_flags: person.behavior_flags,
            wander_duration: person.wander_duration,
            wander_range: person.wander_range,
            linked_obj_id: person.linked_obj_id.map(|h| h as usize),
            bloodlust: person.bloodlust,
            shielded: person.shielded,
            anim: person.anim.clone(),
        })
    } else { None }
}

/// Compatibility accessor: returns collected Unit views.
/// Used by rendering code until fully migrated to pool iterators.
pub fn units(&self) -> Vec<Unit> {
    self.person_ids.iter()
        .filter_map(|&handle| self.unit_view(handle))
        .collect()
}
```

NOTE: `enter_state` and `select_animation` currently take `&mut Unit`. You need to create adapter functions or modify them to take the pool-backed references. The cleanest approach: create a `PersonRef` helper struct that provides mutable access to both header and person data, then adapt the state functions.

Actually, simpler: since person_state.rs functions take `&mut Unit`, and the Unit struct still exists, the fastest migration path is:
1. Extract PersonData + ObjectHeader into a temporary Unit for state function calls
2. Write back the modified fields after the call

But that's wasteful. Better approach: modify `enter_state`, `tick_state`, etc. in person_state.rs to accept a `PersonRef` that wraps (&mut ObjectHeader, &mut PersonData) with the same field access patterns. OR, add a trait that both Unit and the pool-backed view implement.

**RECOMMENDED approach for person_state.rs functions:** Since these functions only access specific fields, create inline helper closures or a struct wrapper:

```rust
/// Mutable view into a person object in the pool.

pub struct PersonRef<'a> {
    pub header: &'a mut ObjectHeader,
    pub person: &'a mut PersonData,
}
```

Then add `impl PersonRef` with accessor methods that match what person_state.rs expects. Modify person_state.rs functions to accept `&mut PersonRef` instead of `&mut Unit`.
